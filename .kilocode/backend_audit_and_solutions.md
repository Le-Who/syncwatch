# Аудит Архитектуры Backend и WebSocket (SyncWatch)

## Введение

В данном документе представлен глубокий архитектурный аудит backend-части проекта SyncWatch, включая WebSocket-сервер (`server.ts`), клиентскую обертку (`lib/socket.ts`) и API-маршруты (`app/api/`). В ходе анализа были выявлены критические уязвимости, узкие места производительности и архитектурные антипаттерны. Для каждой проблемы предложено 5-7 независимых решений с использованием различных парадигм программирования, а также синтезировано финальное бескомпромиссное гибридное решение.

---

## Проблема 1: Монолитное In-Memory состояние и отсутствие горизонтального масштабирования (`server.ts`)

### Описание проблемы

В текущей реализации состояние всех комнат хранится в локальной переменной `const rooms = new Map<string, RoomState>();`. Это классический антипаттерн Stateful Server.

- **Архитектурная ошибка:** Невозможность горизонтального масштабирования (запуск нескольких экземпляров Node.js). Пользователи, подключенные к разным инстансам, не увидят одну и ту же комнату.
- **Узкое место:** Утечки памяти при большом количестве комнат и неэффективный Garbage Collection (очистка только раз в 5 минут).
- **Уязвимость:** При падении процесса Node.js все текущие состояния комнат, не успевшие сохраниться в БД (из-за debounce), безвозвратно теряются.

### 5-7 Альтернативных решений

1.  **Паттерн "Shared State / Distributed Cache" (Redis JSON):** Перенос `Map` в Redis. Все инстансы читают и пишут состояние в единое хранилище.
2.  **Паттерн "Pub/Sub" (Redis Pub/Sub):** Сохранение локального `Map`, но использование Redis Pub/Sub для широковещательной рассылки событий (команд) между всеми инстансами для синхронизации локальных состояний.
3.  **Паттерн "Actor Model" (Cloudflare Durable Objects / PartyKit):** Вынос логики комнат в serverless-акторы. Каждая комната — это изолированный процесс с гарантированным единственным экземпляром (Single Source of Truth) и встроенным in-memory состоянием.
4.  **Паттерн "Event Sourcing" (Kafka / Redis Streams):** Вместо хранения текущего состояния, хранятся только события (Play, Pause, AddItem). Состояние вычисляется "на лету" путем редукции потока событий.
5.  **Паттерн "Sticky Sessions" (Load Balancer):** Использование балансировщика нагрузки (например, Nginx/HAProxy) для жесткой привязки пользователей одной комнаты к одному и тому же инстансу Node.js.
6.  **Паттерн "CRDT" (Conflict-free Replicated Data Type):** Использование библиотек вроде Yjs. Сервер выступает только как ретранслятор (Signaling Server), а состояние децентрализованно хранится и мержится на клиентах.
7.  **Паттерн "Serverless WebSockets" (AWS API Gateway + DynamoDB):** Полный отказ от долгоживущего Node.js сервера в пользу управляемых WebSocket-соединений от облачного провайдера.

### Критический обзор решений

- _Redis JSON_ дает единый источник правды, но требует постоянных сетевых запросов при каждом чихе, что увеличивает latency.
- _Pub/Sub_ быстр (чтение из памяти), но сложен в разрешении конфликтов (Race Conditions), если два инстанса одновременно меняют состояние.
- _Actor Model_ идеален для комнат (как в мультиплеерных играх), но требует жесткой привязки к вендору (Cloudflare).
- _Event Sourcing_ обеспечивает идеальный аудит, но избыточен для простого плеера и сложен в реализации (требует snapshotting).
- _Sticky Sessions_ — это "костыль", который не решает проблему потери данных при падении инстанса.
- _CRDT_ отлично решает проблему масштабирования сервера, но переносит тяжелую логику на клиент и усложняет валидацию прав доступа (авторитарность).

### Синтезированное гибридное решение: "Redis-backed Actor System с локальным кэшированием и Pub/Sub"

**Архитектура:**
Используем гибридный подход. Состояние комнаты хранится в **Redis (Hash/JSON)** как Single Source of Truth. Для минимизации latency каждый Node.js инстанс держит **локальный In-Memory Cache** (как сейчас).
При получении команды от пользователя:

1. Инстанс берет распределенную блокировку (Redis Redlock) на комнату.
2. Обновляет состояние в Redis и в локальном кэше.
3. Публикует событие об обновлении через **Redis Pub/Sub**.
4. Остальные инстансы, получив событие, инвалидируют или обновляют свой локальный кэш и рассылают новые данные своим подключенным WebSocket-клиентам.
   _Преимущества:_ Абсолютная консистентность, возможность запускать 100+ инстансов сервера, защита от потери данных при краше, минимальный пинг для клиентов.

## Проблема 2: Уязвимость SSRF и неэффективный парсинг HTML в API метаданных (`app/api/metadata/route.ts`)

### Описание проблемы

Эндпоинт принимает произвольный `url` от клиента и делает `fetch(url)`.

- **Уязвимость (SSRF - Server-Side Request Forgery):** Злоумышленник может передать внутренние IP-адреса (например, `http://169.254.169.254` для AWS или `http://localhost:3000`) и заставить сервер сканировать внутреннюю сеть.
- **Узкое место (Memory/CPU Exhaustion):** Сервер скачивает весь HTML-документ (который может весить десятки мегабайт) в оперативную память, а затем использует тяжелое регулярное выражение `text.match(/<title[^>]*>([^<]+)<\/title>/i)` для поиска заголовка. Это прямой путь к ReDoS и OOM (Out of Memory).

### 5-7 Альтернативных решений

1.  **Паттерн "Allowlisting" (Белые списки):** Жестко ограничить домены, к которым можно делать запросы (только youtube.com, vimeo.com и т.д.).
2.  **Паттерн "Streaming Parser" (Потоковая обработка):** Использовать Node.js Streams (`htmlparser2`). Читать ответ по чанкам и прерывать соединение (`req.destroy()`) сразу после нахождения тега `<title>`.
3.  **Паттерн "DNS Resolution Hook" (Защита на уровне сети):** Перед `fetch` резолвить домен в IP-адрес и блокировать запрос, если IP принадлежит к приватным подсетям (RFC 1918).
4.  **Паттерн "Byte-Range Requests":** Отправлять HTTP-заголовок `Range: bytes=0-5000`, чтобы скачивать только первые 5 КБ документа, где обычно находится `<title>`.
5.  **Паттерн "Proxy/Egress Gateway":** Пропускать все исходящие запросы через изолированный прокси-сервер (например, Smokescreen), который аппаратно блокирует доступ к внутренним сетям.
6.  **Паттерн "Third-party Delegation":** Использовать внешние сервисы (OpenGraph.io, Iframely) для извлечения метаданных, полностью сняв нагрузку с собственного сервера.
7.  **Паттерн "Headless Browser":** Использовать Puppeteer для рендеринга страницы и извлечения title через DOM API (защищает от SPA-сайтов, где title генерируется JS).

### Критический обзор решений

- _Allowlisting_ слишком ограничивает функционал (пользователи не смогут добавлять прямые ссылки на .mp4 с других сайтов).
- _Streaming Parser_ решает проблему памяти, но не решает SSRF.
- _DNS Hook_ надежно защищает от SSRF, но сложен в реализации из-за возможных DNS Rebinding атак.
- _Byte-Range_ не всегда поддерживается серверами (игнорируют заголовок) и title может быть дальше 5 КБ.
- _Third-party Delegation_ стоит денег и добавляет внешнюю зависимость.
- _Headless Browser_ катастрофически медленный и ресурсоемкий.

### Синтезированное гибридное решение: "SSRF-Safe Streaming Parser с DNS-валидацией"

**Архитектура:**

1. **DNS Resolution & Validation:** Перед запросом парсим URL, извлекаем hostname. Резолвим его в IP. Проверяем IP по черному списку (Bogon IP space: `10.0.0.0/8`, `127.0.0.0/8`, `169.254.0.0/16` и т.д.). Защита от DNS Rebinding обеспечивается фиксацией IP-адреса для последующего `fetch`.
2. **Streaming HTML Parser:** Используем `fetch` с получением `ReadableStream`. Пропускаем поток через легковесный SAX-парсер (например, `htmlparser2`).
3. **Early Abort & Limits:** Устанавливаем жесткий таймаут (3 сек) через `AbortController`. Как только парсер встречает закрывающий тег `</title>`, вызываем `controller.abort()`, мгновенно разрывая TCP-соединение. Дополнительно ставим лимит на чтение (максимум 50 КБ).
   _Преимущества:_ Нулевой риск SSRF, минимальное потребление RAM (O(1)), мгновенный ответ, защита от скачивания гигабайтных файлов.

## Проблема 3: Ненадежная персистентность БД и состояние гонки (`server.ts`)

### Описание проблемы

Функция `persistRoomState` использует простой `setTimeout` для дебаунса.

- **Архитектурная ошибка:** Операции обновления не атомарны. Сначала делается `delete` старых элементов плейлиста, затем `upsert` новых. Если сервер упадет между этими двумя операциями, плейлист будет безвозвратно утерян.
- **Узкое место:** Перед удалением делается `select` существующих ID. Это лишний запрос к БД, замедляющий синхронизацию.
- **Логическая ошибка:** Отсутствие обработки ошибок. Если `supabase.upsert` падает, таймер очищается, и состояние может больше никогда не сохраниться.

### 5-7 Альтернативных решений

1.  **Паттерн "Unit of Work / Database Transaction":** Обернуть все операции (upsert комнаты, delete/upsert плейлиста, upsert playback) в единую транзакцию PostgreSQL (через Supabase RPC).
2.  **Паттерн "Write-Behind Cache" (Message Queue):** Вместо `setTimeout`, отправлять сериализованное состояние в надежную очередь (BullMQ/RabbitMQ). Отдельный воркер будет гарантированно сохранять данные в БД с ретраями.
3.  **Паттерн "Optimistic Concurrency Control" (OCC):** Использовать поле `version`. При сохранении проверять `WHERE version = current_version - 1`. Если не совпадает — кто-то другой уже обновил БД.
4.  **Паттерн "Write-Ahead Log" (WAL):** Записывать каждую команду (JSON) в append-only файл или Redis List _до_ применения в памяти. При краше восстанавливать состояние из лога.
5.  **Паттерн "Graceful Shutdown Hook":** Перехватывать сигналы `SIGTERM`/`SIGINT` процесса Node.js и принудительно выполнять все ожидающие `persistenceTimers` перед завершением работы.
6.  **Паттерн "CQRS":** Разделить модель чтения (In-Memory) и записи. Записывать в БД только дельты (diffs), а не весь массив плейлиста целиком.
7.  **Паттерн "Soft Delete":** Вместо физического удаления (`delete`) элементов плейлиста, помечать их флагом `deleted_at`. Это уберет необходимость делать предварительный `select`.

### Критический обзор решений

- _Транзакции (RPC)_ решают проблему атомарности, но требуют написания PL/pgSQL кода в Supabase.
- _Message Queue_ дает 100% гарантию доставки и ретраи, но усложняет инфраструктуру (нужен Redis).
- _OCC_ хорош для конкурентной записи, но у нас пока один сервер (хотя понадобится при масштабировании).
- _WAL_ избыточен, если мы перейдем на Redis как Source of Truth (Проблема 1).
- _Graceful Shutdown_ обязателен в любом случае, но не спасает от внезапного `kill -9` (OOM).
- _Soft Delete_ решает проблему лишнего `select`, но раздувает таблицу БД.

### Синтезированное гибридное решение: "Transactional Write-Behind Queue с Graceful Shutdown"

**Архитектура:**

1. **Атомарность (Supabase RPC):** Создаем хранимую процедуру (PostgreSQL Function) `sync_room_state`, которая принимает JSON со всем состоянием комнаты и внутри одной транзакции делает все необходимые `INSERT/UPDATE/DELETE`.
2. **Надежная очередь (Write-Behind):** Вместо `setTimeout` используем in-memory очередь с поддержкой персистентности (или Redis BullMQ). При изменении состояния задача кладется в очередь. Воркер берет задачу и вызывает RPC. Если БД недоступна — задача возвращается в очередь (Exponential Backoff Retry).
3. **Graceful Shutdown:** При получении `SIGTERM` сервер перестает принимать новые WebSocket соединения, дожидается опустошения очереди записи в БД и только потом завершает процесс.
   _Преимущества:_ Гарантия консистентности данных (ACID), отсутствие блокировок Event Loop, устойчивость к падениям БД и самого сервера.

## Проблема 4: Утечка ресурсов из-за неотменяемых промисов (`app/api/youtube/...`)

### Описание проблемы

В маршрутах поиска и плейлистов YouTube используется конструкция `Promise.race([searchPromise, timeoutPromise])`.

- **Узкое место / Утечка ресурсов:** Если срабатывает таймаут (5 секунд), клиент получает ответ 500. Однако `searchPromise` (вызов библиотеки `yt-search`) **продолжает выполняться в фоне**. Он потребляет CPU, держит открытые TCP-сокеты и расходует оперативную память. При DDoS-атаке медленными запросами сервер быстро исчерпает лимит соединений (Socket Exhaustion).

### 5-7 Альтернативных решений

1.  **Паттерн "Cancellation Token" (AbortController):** Передать `AbortSignal` внутрь функции `yts`, чтобы она могла прервать HTTP-запрос на уровне сетевого стека.
2.  **Паттерн "Process Isolation" (Child Process):** Запускать парсинг в отдельном процессе (`child_process.fork`). При таймауте делать жесткий `process.kill()`.
3.  **Паттерн "Worker Threads":** Использовать потоки Node.js. При таймауте вызывать `worker.terminate()`, что мгновенно освободит ресурсы.
4.  **Паттерн "Adapter / Facade" (Официальный API):** Отказаться от веб-скрапинга (`yt-search`) в пользу официального YouTube Data API v3, который работает в разы быстрее и поддерживает нативные таймауты.
5.  **Паттерн "Cache-Aside" (Redis/In-Memory Cache):** Кэшировать результаты поиска по ключу (строка запроса) на 1-2 часа. Это кардинально снизит количество вызовов тяжелой функции.
6.  **Паттерн "Circuit Breaker":** Если `yts` начинает часто падать по таймауту (например, YouTube включил капчу), временно блокировать все запросы (Fail Fast), чтобы не усугублять перегрузку сервера.
7.  **Паттерн "Bulkhead Isolation":** Ограничить максимальное количество одновременных вызовов `yts` (например, не более 10 параллельных промисов) с помощью семафора. Остальные запросы сразу получают 429 Too Many Requests.

### Критический обзор решений

- _AbortController_ — идеальный вариант, но библиотека `yt-search` его **не поддерживает** из коробки.
- _Child Process / Worker Threads_ решают проблему утечки, но создание потока/процесса на каждый HTTP-запрос — это огромный оверхед по CPU и RAM.
- _Официальный API_ требует ключа и имеет квоты, но это единственный production-ready путь.
- _Кэширование_ обязательно, но не решает проблему первого долгого запроса (Cache Miss).
- _Circuit Breaker_ и _Bulkhead_ защищают сервер, но ухудшают UX при пиковых нагрузках.

### Синтезированное гибридное решение: "Официальный API + Многоуровневое кэширование + Circuit Breaker"

**Архитектура:**

1. **Миграция на YouTube Data API v3:** Замена нестабильного скрапера на официальный REST API. Использование нативного `fetch` с передачей `AbortSignal` для честного прерывания TCP-соединения при таймауте.
2. **Многоуровневый кэш (Stale-While-Revalidate):**
   - L1: In-Memory LRU Cache (быстрый ответ).
   - L2: Redis Cache (распределенный).
     Если данные устарели, клиенту отдается старый кэш, а в фоне запускается обновление (SWR).
3. **Circuit Breaker & Fallback:** Если официальный API исчерпал квоту (403) или недоступен, Circuit Breaker переключает маршрутизацию на резервный метод (тот самый `yt-search`, но обернутый в Worker Thread с жестким `terminate()` по таймауту для безопасности).
   _Преимущества:_ Максимальная производительность, нулевая утечка ресурсов, отказоустойчивость при исчерпании квот YouTube.

## Проблема 5: Небезопасная аутентификация WebSocket и обход Rate Limiting

### Описание проблемы

- **Уязвимость (Spoofing):** В `server.ts` авторизация команд происходит путем сравнения `participant.sessionToken !== sessionToken`. Токен генерируется на клиенте (`randomUUID()`) или сервере и передается в открытом виде. Любой, кто узнает токен (XSS, сниффинг), может отправлять команды от имени владельца комнаты.
- **Уязвимость (Rate Limit Bypass):** В API используется `request.headers.get("x-forwarded-for")`. Злоумышленник может отправить заголовок `X-Forwarded-For: 1.2.3.4` и обойти лимиты, так как Next.js слепо доверяет этому заголовку без проверки доверенных прокси.

### 5-7 Альтернативных решений

1.  **Паттерн "Token-Based Auth" (JWT):** Сервер генерирует криптографически подписанный JWT (HMAC SHA-256) при входе в комнату. Клиент отправляет его при каждом WebSocket-сообщении. Подделать подпись невозможно.
2.  **Паттерн "Session-Based Auth" (HTTP-Only Cookies):** Использовать куки для хранения сессии. WebSocket автоматически подхватывает куки при handshake. Защищает от XSS.
3.  **Паттерн "HMAC Request Signing":** Клиент и сервер обмениваются секретом при handshake. Клиент подписывает каждый payload (хэш от данных + секрет). Защищает от подмены тела команды (Tampering).
4.  **Паттерн "Distributed Rate Limiting" (Redis Token Bucket):** Хранить счетчики запросов не в памяти одного инстанса, а в Redis.
5.  **Паттерн "Reverse Proxy Trust Configuration":** Настроить сервер так, чтобы он брал IP только из сокета (`req.socket.remoteAddress`), а `x-forwarded-for` парсил только если запрос пришел от доверенного IP (например, Cloudflare).
6.  **Паттерн "WebSocket Connection Tracking":** Ограничить количество активных сокетов с одного IP (не более 5) на уровне `io.engine.generateId`.
7.  **Паттерн "Proof of Work" (PoW):** Заставлять клиента вычислять криптографическую задачу (например, найти хэш с нулями) перед подключением к WebSocket, чтобы сделать DDoS экономически невыгодным для атакующего.

### Критический обзор решений

- _JWT_ отлично подходит для stateless авторизации, но его сложно инвалидировать до истечения срока жизни.
- _HTTP-Only Cookies_ — золотой стандарт безопасности, но требует настройки CORS и `withCredentials` в Socket.io.
- _HMAC Signing_ избыточен для плеера (это не банковское приложение) и сильно нагружает CPU.
- _Redis Rate Limiting_ обязателен для распределенной системы.
- _Proxy Trust_ — критически важная настройка безопасности сети, без нее любой Rate Limit бесполезен.
- _PoW_ ухудшает UX (задержка при входе) и тратит батарею мобильных устройств.

### Синтезированное гибридное решение: "Secure Handshake с JWT и Distributed Rate Limiting"

**Архитектура:**

1. **Аутентификация (JWT + HTTP-Only Cookie):** При создании/входе в комнату через REST API, сервер выдает `HTTP-Only`, `Secure`, `SameSite=Strict` cookie с подписанным JWT (содержит `roomId`, `participantId`, `role`). WebSocket handshake (`io.on("connection")`) валидирует этот JWT. Токены больше не передаются в payload команд.
2. **Защита от подмены (Spoofing):** Поскольку сокет жестко привязан к проверенному `participantId` на этапе handshake, сервер игнорирует любые попытки клиента передать чужой ID в командах.
3. **Надежный Rate Limiting:**
   - Настройка Next.js/Node.js на доверие только известным прокси (Cloudflare IP ranges).
   - Использование алгоритма **Sliding Window Log** в Redis для ограничения как HTTP-запросов, так и частоты WebSocket-команд (защита от спама кнопкой Play/Pause).
   - Ограничение на уровне Socket.io: не более 3 одновременных подключений с одного реального IP.
     _Преимущества:_ Полная защита от XSS (токен в недоступной куке), невозможность подделки команд, надежная защита от DDoS и брутфорса API.
