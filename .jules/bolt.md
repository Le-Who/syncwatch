## 2024-05-24 - setTimeout vs requestAnimationFrame for Playback Sync
**Learning:** While `requestAnimationFrame` is great for immediate per-frame visual updates (like scrubbing), using it as a general timer replacement for long intervals (e.g. >200ms) creates a "busy-wait" loop. The callback fires ~60 times a second just to check if enough time has passed, which unnecessarily burns CPU cycles and drains battery, completely degrading performance.
**Action:** For non-visual background syncing or adaptive intervals that exceed a single frame duration, stick to `setTimeout` or `setInterval` to allow the CPU to actually rest between executions. Use `requestAnimationFrame` strictly when you need to manipulate the DOM or Canvas on every single frame.

## 2024-05-24 - Global Store Subscriptions and Re-renders
**Learning:** Subscribing to an entire deeply-nested object from a global store (e.g., `useStore(s => s.room)`) causes the component to re-render every time *any* property inside that object changes. In a real-time app, properties like `playback.baseTimestamp` or `serverClockOffset` update constantly, triggering massive cascading re-renders across the entire React tree.
**Action:** Always use targeted selectors or `useShallow` (e.g., `useStore(useShallow(s => s.room?.participants))`) to only subscribe to the specific slice of data the component actually renders. If a rapidly changing value is only needed for imperative calculations (like math inside a `setInterval`), read it directly via `useStore.getState()` instead of making the component reactive to it.
